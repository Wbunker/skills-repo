# Documenting the UserRepository Security Constraint

This is a multi-layer documentation problem. The constraint needs to be enforced at every level where code gets written: in the AI context file (CLAUDE.md/AGENTS.md), in an ADR explaining why the constraint exists, in the code itself, and in the repository structure. A constraint this serious — a potential security vulnerability — deserves all of those layers working together.

---

## Why one document is not enough

Human engineers and AI tools find documentation through different paths:

- A new engineer onboarding reads the README and architecture docs
- An AI coding assistant reads CLAUDE.md/AGENTS.md at session start
- A developer browsing the codebase reads docstrings and inline comments as they work
- Anyone asking "why does this constraint exist?" needs an ADR

If you put the rule only in one place, you'll miss one or more of these paths. A security constraint that bypasses row-level security is exactly the situation that warrants redundant, layered documentation.

---

## Layer 1: CLAUDE.md / AGENTS.md — the AI context file

This is the most important layer for AI tools. CLAUDE.md is loaded at the start of every session, before any code is written. Put the constraint in a dedicated "Security Constraints" or "Critical Constraints" section near the top of the file — not buried at the bottom.

```markdown
## Security Constraints

- NEVER write direct SQL queries against the `users` table. ALWAYS use
  `UserRepository` instead. Direct queries bypass row-level security (tenant_id
  filtering) and are a security vulnerability. See ADR-0005.
  See @docs/decisions/0005-user-repository-required.md
```

Key principles applied here:
- The constraint is stated as NEVER/ALWAYS (imperative; hard to misread)
- The positive alternative is given immediately ("use UserRepository instead") — rules without alternatives are less effective
- The reason is stated inline (RLS bypass = security vulnerability) — this is load-bearing because it transforms a rule into a policy an AI can reason about
- The ADR is referenced so the full context is one click away
- The `@` import syntax lets Claude load the ADR when working on related tasks without bloating every session

If your team uses multiple AI tools (Copilot, Cursor, Gemini CLI), maintain this constraint in `AGENTS.md` as the cross-tool universal file, using the same content without the Claude-specific `@` import syntax.

---

## Layer 2: ADR — the "why"

An ADR prevents the constraint from being "improved away." Without an ADR, a future engineer (or AI) might look at UserRepository, decide it's an unnecessary abstraction, and refactor around it — reasoning that direct queries are simpler. The ADR makes the security rationale durable and explicit.

Save this as `docs/decisions/0005-user-repository-required.md`:

```markdown
# ADR-0005: All user table access must go through UserRepository

**Status**: Accepted
**Date**: [date]
**Deciders**: [team handles]

## Context

This is a multi-tenant system. Every row in the `users` table belongs to a
specific tenant, identified by the `tenant_id` column. If any query reads or
writes user data without filtering by `tenant_id`, it can access or modify
another tenant's data — a serious data isolation and security vulnerability.

UserRepository was built to enforce this invariant. Every method in
UserRepository adds the `tenant_id` filter automatically, derived from the
authenticated request context. Direct SQL queries against the `users` table
do not have this enforcement.

This constraint has been violated before by direct queries written without
awareness of the multi-tenancy requirement, including queries generated by
AI coding tools that did not have this context.

## Decision

All access to the `users` table — reads, writes, updates, deletes — MUST go
through `UserRepository`. Direct SQL queries against `users` are prohibited,
regardless of whether the query looks "simple" or "read-only."

This rule applies to:
- Application code
- Scripts and one-off utilities
- Migration scripts (use the migration-specific utilities in `scripts/db/`)
- Code generated by AI coding assistants

## Consequences

**Positive:**
- Tenant data isolation is guaranteed by construction
- No single developer or AI session can introduce a tenant isolation bug by
  writing an unfiltered query

**Negative / trade-offs:**
- Adds a layer of indirection for queries that are genuinely cross-tenant
  (admin operations). These must use `UserRepository.asAdmin()` explicitly,
  which makes the elevated-permission query visible in code review.

**Alternatives rejected:**
- Database-level RLS (PostgreSQL row security policies): Evaluated but
  rejected because it creates invisible filtering that is harder to test,
  harder to debug, and creates false confidence in raw queries.
- Linting rule only: A linter can be bypassed or suppressed; the documented
  constraint + code review is a more durable enforcement mechanism.
```

## Layer 3: Code-level documentation — inline with the work

Documentation that lives with the code is read automatically as engineers and AI tools work in the repository. Two high-value places:

### The UserRepository class itself

Add a prominent module-level docstring that states the security role of this class:

```java
/**
 * All access to the users table MUST go through this repository.
 *
 * <p>This repository enforces row-level security by automatically appending
 * {@code WHERE tenant_id = :currentTenantId} to every query, derived from
 * the authenticated request context ({@link TenantContext}).
 *
 * <p><strong>Do not write direct SQL against the users table.</strong>
 * Direct queries bypass tenant isolation and are a security vulnerability.
 * See ADR-0005 for the full rationale.
 *
 * <p>For admin operations that legitimately span tenants, use
 * {@link #asAdmin()} — this makes the elevated-permission access explicit
 * and visible in code review.
 */
public class UserRepository { ... }
```

This docstring is the first thing an AI reads when it opens the file. It will use the context when generating code that touches UserRepository.

### The users table schema or migration file

Add a comment at the top of the migration or schema file:

```sql
-- IMPORTANT: Never query this table directly in application code.
-- All access must go through UserRepository (see src/.../UserRepository.java).
-- Direct queries bypass tenant_id row-level security. See ADR-0005.
CREATE TABLE users (
    id          UUID PRIMARY KEY,
    tenant_id   UUID NOT NULL REFERENCES tenants(id),
    ...
);
```

## Layer 4: Subdirectory CLAUDE.md (optional, high value for large repos)

If your data access layer is large enough to have its own directory (e.g., `src/repositories/` or `src/data/`), add a subdirectory CLAUDE.md there:

```markdown
# Data Access Layer

All database access is mediated through repository classes.

## Security constraint: users table

NEVER write direct SQL against the `users` table.
ALWAYS use `UserRepository`. Direct queries bypass tenant_id filtering
(row-level security). This is a security vulnerability. See ADR-0005.

## Pattern

Every repository takes a `TenantContext` in its constructor or via
`@Autowired`. The context provides the current tenant_id for all queries.
```

Claude Code loads subdirectory CLAUDE.md files when working in that directory context, making this a targeted, high-signal reinforcement exactly where the dangerous work happens.

## Layer 5: Enforcement mechanisms (documentation is not enough alone)

Documentation deters mistakes; it does not prevent them. For a security constraint, add at least one enforcement mechanism alongside the documentation:

**Static analysis**: Write a custom lint rule or ArchUnit test that fails if any class outside `UserRepository` contains a SQL string referencing the `users` table. This creates a CI gate:

```java
// ArchUnit example (Java)
@Test
void no_direct_sql_against_users_table() {
    noClasses()
        .that().resideOutsideOfPackage("..repository..")
        .should().containAnyFieldsOfType(String.class)
        .check(/* classes containing "SELECT.*FROM users" */);
}
```

The exact implementation depends on your stack, but the principle is: make the violation fail CI, not just violate a written rule.

**Reference the enforcement in CLAUDE.md**: Once the lint rule exists, mention it — this tells AI tools that the constraint is machine-verified, not just aspirational:

```markdown
## Security Constraints

- NEVER write direct SQL against the `users` table. ALWAYS use `UserRepository`.
  Bypassing this is a tenant isolation vulnerability (ADR-0005).
  This is enforced by the `UserTableDirectAccessTest` ArchUnit rule.
```

## Summary: the full documentation stack

| Layer | Artifact | Audience | Why it matters |
|-------|----------|----------|----------------|
| AI context | CLAUDE.md / AGENTS.md | AI tools at session start | Loaded before any code is written |
| Rationale | ADR-0005 | Future engineers, AI tools, code reviewers | Prevents the constraint being "refactored away" |
| Class docstring | UserRepository Javadoc/docstring | AI tools reading the file, developers | Read automatically when the file is opened |
| Schema comment | users table migration | Developers and AI tools writing migrations | Present at the exact location of potential bypass |
| Subdirectory context | `src/repositories/CLAUDE.md` | AI tools working in the data layer | Targeted reinforcement where the risk is highest |
| Enforcement | ArchUnit / lint rule | CI pipeline | Makes violation a build failure, not just a policy |

The documentation layers change behavior; the enforcement layer catches the cases documentation misses. For a constraint with security implications, you want both.
