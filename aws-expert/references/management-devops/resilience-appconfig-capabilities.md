# Resilience & AppConfig — Capabilities Reference

For CLI commands, see [resilience-appconfig-cli.md](resilience-appconfig-cli.md).

## AWS Infrastructure Composer

**Purpose**: Visual drag-and-drop canvas for designing and composing serverless and cloud-native applications; bidirectionally syncs with CloudFormation/SAM templates so the visual diagram and the IaC code are always in sync.

### Core Concepts

| Concept | Description |
|---|---|
| **Canvas** | Visual workspace where you drag AWS resources and connect them with event flows, triggers, and permissions |
| **Resource card** | Visual representation of an AWS resource on the canvas; configures properties in a side panel |
| **Connection** | A drawn edge between two resource cards; translates into IAM policies, event source mappings, or resource references in the template |
| **Template sync** | Bidirectional sync between the canvas state and the underlying CloudFormation/SAM YAML; edits to either side are reflected in the other |
| **Enrich mode** | Composer enhances an existing CloudFormation/SAM template with visual layout without modifying template logic |
| **Application Composer (VS Code extension)** | The same canvas experience available inside VS Code via the AWS Toolkit extension |

### Supported Template Formats

| Format | Use case |
|---|---|
| **AWS SAM (YAML)** | Preferred for serverless; `AWS::Serverless::Function`, `AWS::Serverless::Api`, `AWS::Serverless::StateMachine` |
| **CloudFormation (YAML/JSON)** | Full resource coverage; any CloudFormation resource type |

### Key Workflow

1. Open Infrastructure Composer in the AWS Console (or VS Code)
2. Drag resources from the panel onto the canvas (Lambda, API Gateway, DynamoDB, SQS, SNS, Step Functions, S3, etc.)
3. Draw connections between resources — Composer generates IAM policies and event triggers automatically
4. Configure resource properties in the right-side properties panel
5. Download or deploy the generated SAM/CloudFormation template directly

### Key Patterns

- **Prototyping** — quickly visualize a serverless architecture before writing any YAML; useful for team design reviews
- **Template import** — import an existing SAM or CloudFormation template to get a visual diagram and then continue editing both visually and as code
- **SAM CLI integration** — export the SAM template from Composer and deploy via `sam build && sam deploy`
- **VS Code workflow** — use the Application Composer panel in VS Code for a local-first IaC authoring experience without leaving the editor

## AWS Resilience Hub

**Purpose**: Continuously assess, validate, and track application resilience against user-defined RTO and RPO targets; surfaces gaps and prescriptive recommendations before an outage occurs.

### Core Concepts

| Concept | Description |
|---|---|
| **Application** | A logical grouping of AWS resources that form a single resilience boundary; defined by importing from CloudFormation stacks, Resource Groups, Terraform state, or AppRegistry |
| **AppComponent** | A group of related resources that fail as a unit (e.g., primary RDS instance + read replica); resources in the same AppComponent share an RTO/RPO target |
| **Resiliency policy** | Defines the acceptable RTO and RPO targets for each disruption type; associated with an application |
| **Assessment** | A point-in-time evaluation of the application against its resiliency policy; produces a compliance status and resiliency score |
| **Resiliency score** | 0–100 score reflecting adherence to alarm configuration, SOPs, and resilience testing recommendations |
| **SOP (Standard Operating Procedure)** | Prescriptive recovery runbook generated by Resilience Hub for restoring an application after a disruption |
| **Drift detection** | Continuous monitoring that flags when the application's deployed resources diverge from the last assessed version |

### Disruption Types

| Disruption Type | Scope | Example |
|---|---|---|
| **Application disruption** | Application layer | Bad code deployment, data corruption, configuration error |
| **Infrastructure disruption** | Component failure | Instance crash, container restart |
| **AZ disruption** | Availability Zone outage | Single AZ goes unavailable |
| **Region disruption** | Regional outage | Full region outage; requires cross-region failover |

### Compliance Status

| Status | Meaning |
|---|---|
| **PolicyMet** | Estimated RTO/RPO is within policy targets for all AppComponents |
| **PolicyBreached** | One or more AppComponents cannot meet RTO/RPO targets |
| **NotAssessed** | Application has not been assessed yet |
| **ChangesDetected** | Resources have changed since last assessment; re-assessment required |

### Recommendation Types

| Type | Description |
|---|---|
| **Alarm recommendations** | CloudWatch alarms to add for detecting outages (e.g., CPU, error rates, latency) |
| **SOP recommendations** | AWS Systems Manager Automation runbooks to prepare for recovery |
| **Test recommendations** | AWS FIS experiment templates to inject failures and verify recovery |
| **Component recommendations** | Architectural changes to improve resilience (e.g., add Multi-AZ, add read replicas) |

### Key Patterns

- **CloudFormation import** — define the application by pointing at existing CloudFormation stacks; Resilience Hub automatically discovers all included resources
- **Multi-stack applications** — import multiple stacks into a single Resilience Hub application to model complex, multi-tier systems
- **FIS integration** — Resilience Hub generates ready-to-use FIS experiment templates; run them to validate that alarms fire and SOPs restore the application within RTO/RPO
- **Drift detection** — configure EventBridge notifications so teams are alerted when resource changes may degrade resilience without re-assessment
- **myApplications integration** — Resilience Hub applications appear in the AWS myApplications view for unified operational health across the console
- **Policy tiers** — define multiple policies (e.g., Tier 1: RTO 1h / RPO 1h; Tier 2: RTO 4h / RPO 4h) and assign them to different applications based on criticality

### CDK Integration

```python
# aws_cdk.aws_resiliencehub (L1 constructs)
from aws_cdk import aws_resiliencehub as resiliencehub

policy = resiliencehub.CfnResiliencyPolicy(
    self, "TierOnePolicy",
    policy_name="Tier1-RTO1h-RPO1h",
    tier="MissionCritical",
    policy={
        "AZ": resiliencehub.CfnResiliencyPolicy.FailurePolicyProperty(
            rto_in_secs=3600,
            rpo_in_secs=3600,
        ),
        "Hardware": resiliencehub.CfnResiliencyPolicy.FailurePolicyProperty(
            rto_in_secs=3600,
            rpo_in_secs=3600,
        ),
        "Software": resiliencehub.CfnResiliencyPolicy.FailurePolicyProperty(
            rto_in_secs=3600,
            rpo_in_secs=3600,
        ),
        "Region": resiliencehub.CfnResiliencyPolicy.FailurePolicyProperty(
            rto_in_secs=86400,
            rpo_in_secs=86400,
        ),
    },
)
```

## AWS AppConfig

**Purpose**: Deploy and validate application configuration changes (feature flags, dynamic settings, allow/block lists) to running applications with automatic rollback on failure — without a code deployment.

### Core Concepts

| Concept | Description |
|---|---|
| **Application** | A namespace/organizational container for related configuration profiles; maps to a service or application boundary |
| **Environment** | A logical deployment target (e.g., `dev`, `staging`, `production`); holds the currently active configuration and deployment history |
| **Configuration profile** | Defines the configuration source (where the data lives) and type (feature flag or freeform) |
| **Hosted configuration version** | A version of configuration data stored directly in AppConfig; supports JSON, YAML, or text |
| **Deployment strategy** | Controls the rollout speed: how quickly the new configuration spreads to all clients (bake time, growth factor, growth type) |
| **Deployment** | The act of applying a configuration version to an environment using a deployment strategy |
| **Extension** | Custom logic (Lambda-backed) triggered at lifecycle hooks during a deployment (e.g., `ON_DEPLOYMENT_START`, `ON_DEPLOYMENT_COMPLETE`, `ON_DEPLOYMENT_ROLLED_BACK`) |
| **AppConfig Agent** | A sidecar/daemon that runs alongside applications; caches configuration locally and polls for changes; reduces API calls and cost |

### Configuration Profile Types

| Type | Description |
|---|---|
| **Feature flag** | Structured JSON backed by AppConfig's feature flag schema; provides type safety and a visual editor in the console |
| **Freeform** | Arbitrary JSON, YAML, or text; source can be AppConfig hosted store, S3, SSM Parameter Store, or Secrets Manager |

### Validators

| Validator Type | How it works |
|---|---|
| **JSON Schema** | AppConfig validates the configuration JSON against a provided JSON Schema before any deployment proceeds |
| **Lambda** | AppConfig calls a Lambda function with the configuration data; the function returns pass/fail; enables semantic validation (e.g., ensure feature flag percentages sum to 100) |

### Deployment Strategy Parameters

| Parameter | Description |
|---|---|
| **Growth factor** | Percentage of clients receiving the new configuration in each interval |
| **Growth type** | `LINEAR` (same % each step) or `EXPONENTIAL` (doubling each step) |
| **Deployment duration** | Total minutes for the rollout |
| **Bake time** | Minutes to wait after 100% rollout before marking deployment complete; CloudWatch alarms are monitored during bake time |

### AWS-Managed Deployment Strategies

| Strategy | Behavior |
|---|---|
| `AppConfig.AllAtOnce` | Immediate 100% rollout; no bake time |
| `AppConfig.Linear50PercentEvery30Seconds` | 50% every 30s; completes in 1 min |
| `AppConfig.Canary10Percent20Minutes` | 10% immediately, then 90% after 20-min bake |

### Automatic Rollback

AppConfig monitors a CloudWatch alarm associated with an environment. If the alarm triggers during or after deployment, AppConfig automatically rolls back to the previous configuration version and marks the deployment as rolled back.

### AppConfig Agent

The AppConfig Agent is available as:
- An AWS Lambda extension (add as a layer; retrieve config via `http://localhost:2772`)
- An Amazon ECS sidecar container
- An EC2 systemd service

```bash
# Retrieve configuration from AppConfig Agent (from within application)
curl http://localhost:2772/applications/MyApp/environments/production/configurations/feature-flags
```

### Key Patterns

- **Feature flags with staged rollout** — use `Canary10Percent20Minutes` strategy with a CloudWatch alarm on error rate; if errors spike during the canary phase, AppConfig auto-rolls back
- **Lambda extension** — add the AppConfig Agent Lambda layer to avoid per-invocation `GetConfiguration` API calls; the agent caches and handles TTL-based refresh
- **Secrets Manager source** — point a freeform configuration profile at a Secrets Manager secret to enable AppConfig-managed rotation with deployment strategies and rollback
- **Extensions for notifications** — use `ON_DEPLOYMENT_ROLLED_BACK` extension hook to trigger an SNS notification or PagerDuty alert when a configuration deployment is automatically reversed
- **Multi-environment promotion** — deploy to `dev` → `staging` → `production` as separate AppConfig deployments; each environment can have its own CloudWatch alarm monitor

### CDK Integration

```python
from aws_cdk import aws_appconfig as appconfig

app = appconfig.Application(self, "MyApp", application_name="my-service")

env = appconfig.Environment(self, "Prod",
    application=app,
    environment_name="production",
    monitors=[appconfig.Monitor.from_cloud_watch_alarm(error_rate_alarm)],
)

profile = appconfig.HostedConfiguration(self, "FeatureFlags",
    application=app,
    content=appconfig.ConfigurationContent.from_inline_json(
        '{"enableNewCheckout":{"enabled":false}}'
    ),
    type=appconfig.ConfigurationType.FEATURE_FLAGS,
)

appconfig.DeploymentStrategy(self, "Canary",
    deployment_strategy_name="Canary10Percent20Minutes",
    rollout_strategy=appconfig.RolloutStrategy.canary(
        growth_factor=10,
        final_bake_time=cdk.Duration.minutes(20),
    ),
)
```
